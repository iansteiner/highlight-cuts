## Key Decisions

### Two-Tier Structure

**Decision**: Separate user and developer documentation into distinct directories.

**Rationale**:
- Users don't need technical architecture details
- Developers need depth, users need simplicity
- Clearer navigation and discoverability
- Easier to maintain appropriate tone/depth for each audience

### AI-Optimized Documentation

**Decision**: Explicitly optimize documentation for AI consumption.

**Rationale**:
- AI assistants are primary consumers of docs (answering questions, suggesting code)
- Structured format improves AI comprehension
- Benefits humans too (clearer, more scannable)
- Front-loaded info helps AI quickly find relevant content

### Always Create Session Summaries

**Decision**: Create summaries for every session, including doc changes.

**Rationale**:
- Provides complete development history
- Helps future AI sessions understand context
- User can choose not to commit if not needed
- Better to have and not need than need and not have

### Mermaid Over ASCII

**Decision**: Use Mermaid diagrams instead of ASCII art.

**Rationale**:
- Renders beautifully in GitHub, VS Code, most viewers
- Text-based, version control friendly
- AI can understand and suggest modifications
- Professional appearance
- Accessible to screen readers

### Casual Tone

**Decision**: Use casual, friendly tone throughout documentation.

**Rationale**:
- More approachable for newcomers
- Matches open-source community culture
- Still maintains technical accuracy
- AI can generate in consistent voice
